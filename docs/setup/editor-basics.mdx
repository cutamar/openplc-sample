This page documents a small test program created in OpenPLC Editor before implementing the final temperature control logic.

The purpose of this step is to verify that the editor works, variables can be created, Structured Text logic can be written, and the debugger can be used to observe variable states.

## Goal of the Test Program

The test program uses simple Boolean logic to simulate a start/stop control circuit.

This is not the final temperature control logic. It is only a small test program to become familiar with:

- creating a project in OpenPLC Editor
- selecting Structured Text as the programming language
- defining Boolean variables
- writing Structured Text
- understanding cyclic PLC execution
- checking variable states in the debugger

## Creating the First Project

After starting OpenPLC Editor, create a new project from the start screen.

Click **New Project**.

![OpenPLC Editor start screen with New Project button](/setup/assets/editor-new-project.png)

Select **PLC Project** and continue with **Next**.

![Create new PLC project](/setup/assets/editor-select-plc-project.png)

Enter a project name and choose an empty directory for the project files.

For this test, the project was named:

```text
st-learning
```

![Project name and directory selection](/setup/assets/editor-project-name-directory.png)

Next, choose the language for the base program.

For this project, select:


The cyclic task time was set to:

```text

```

Then click **Create Project**.

## Task Cycle Time

When creating the program, the task cycle time defines how often the PLC program is executed.

For example, a task cycle time of `20 ms` means that the PLC executes the program approximately every 20 milliseconds.

In this project, `20 ms` is a reasonable choice because:

- the logic is very small
- the system does not need extremely fast control
- it gives responsive behavior without being unnecessarily aggressive

## Other Possible Cycle Times

Different projects may use different cycle times.

| Cycle Time | Example Use Case |
|---:|---|
| `5 ms` | Very fast machine control or motion-related logic |
| `10 ms` | Fast digital input/output logic |
| `20 ms` | General PLC logic and small control tasks |
| `100 ms` | Slow processes such as temperature monitoring |
| `500 ms` or more | Very slow monitoring or background tasks |

For this assignment, `20 ms` is a safe default for testing. For a real temperature process, even a slower cycle time could be enough because temperature usually changes slowly.

## Test Program Description

The test program controls a simulated motor output.

It has:

| Variable | Type | Description |
|---|---|---|
| `Start_Button` | `BOOL` | Simulated start button |
| `Stop_Button` | `BOOL` | Simulated stop button |
| `Motor_Output` | `BOOL` | Simulated motor output |

When `Start_Button` is pressed, the motor output turns on.

When `Start_Button` is released, the motor output stays on.

When `Stop_Button` is pressed, the motor output turns off.

## Defining the Variables

The variables were defined in the `main` program as local Boolean variables.

| Name | Class | Type | Initial Value |
|---|---|---|---|
| `Start_Button` | Local | `BOOL` | `FALSE` |
| `Stop_Button` | Local | `BOOL` | `FALSE` |
| `Motor_Output` | Local | `BOOL` | `FALSE` |

In OpenPLC Editor, the variables can be added in the variable table above the Structured Text code area.

![Boolean variables and Structured Text code in OpenPLC Editor](/setup/assets/editor-boolean-variables-code.png)

## Structured Text Logic

The test logic is:

```pascal
Motor_Output := (Start_Button OR Motor_Output) AND NOT Stop_Button;
```

## Explanation

The expression uses `OR Motor_Output` to create a latch.

This means that after `Motor_Output` becomes `TRUE`, it keeps itself on in the next PLC cycles.

The `Stop_Button` resets the latch. When `Stop_Button` is `TRUE`, the expression becomes `FALSE`, and the motor output turns off.

In other words:

- `Start_Button` turns the output on
- `Motor_Output` keeps the output on after the start button is released
- `Stop_Button` turns the output off

## Test Sequence

| Step | Action | `Start_Button` | `Stop_Button` | Expected `Motor_Output` |
|---:|---|---|---|---|
| 1 | Initial state | `FALSE` | `FALSE` | `FALSE` |
| 2 | Press start | `TRUE` | `FALSE` | `TRUE` |
| 3 | Release start | `FALSE` | `FALSE` | `TRUE` |
| 4 | Press stop | `FALSE` | `TRUE` | `FALSE` |
| 5 | Release stop | `FALSE` | `FALSE` | `FALSE` |

## How the Button Press Was Simulated

Because no physical buttons were connected, button presses were simulated by manually changing Boolean variable values in the debugger.

For example:

- pressing the start button means setting `Start_Button` to `TRUE`
- releasing the start button means setting `Start_Button` back to `FALSE`
- pressing the stop button means setting `Stop_Button` to `TRUE`

The debugger shows the current values of the variables while the program is running.

![Debugger showing Boolean variables](/setup/assets/editor-debugger-variables.png)
